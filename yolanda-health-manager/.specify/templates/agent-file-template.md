## Constitutional Principles (NON-NEGOTIABLE)

All development MUST comply with these 5 core principles:

1. **Security & Privacy First** - Local-only data, macOS Keychain, SQLCipher encryption
2. **Medical Data Integrity** - Preserve raw FHIR JSON, never modify DICOM, store reference ranges
3. **Offline-First Architecture** - Works without network, graceful API degradation
4. **Accessibility & Caregiver Focus** - WCAG 2.1 AA, plain English, keyboard shortcuts
5. **Apple Silicon Optimization** - Native ARM64, <2GB RAM, <3s startup on M1 Max

**Violations of Principles I-III are REJECTED regardless of functionality.**

See `.specify/memory/constitution.md` for complete details and enforcement rules.

## Medical Data Operations

```bash

npm run fhir:validate          # Check FHIR R4 compliance
npm run fhir:test-oauth        # Test OAuth2 flow in sandbox
```

### DICOM Processing
```bash
npm run dicom:import <dir>     # Import DICOM directory (scans metadata, updates DB)
npm run dicom:validate         # Check DICOM file integrity
python src/python/dicom_loader.py scan /path/to/dicom  # Scan directory structure
python src/python/dicom_loader.py metadata /path/to/file.dcm  # Extract metadata
```

## Code Style

### TypeScript/JavaScript
- **Strict mode enabled**: No `any` types in production code
- **ESLint + Prettier**: Enforced via pre-commit hooks
- **Naming**: camelCase for variables/functions, PascalCase for React components
- **Imports**: Absolute paths via tsconfig paths (`@/components`, `@/services`)
- **Comments**: JSDoc for all public functions (especially FHIR parsers, medical logic)

### Python
- **Type hints**: Required for all function signatures
- **Black formatting**: 88 character line limit
- **Docstrings**: Google style for all modules and public functions
- **Medical data**: Never print PHI/PII to stdout (use sanitization utility)

### SQL (SQLite)
- **Schema**: Use migrations (never direct ALTER TABLE in code)
- **Queries**: Parameterized queries only (no string interpolation)
- **Indexes**: Index all foreign keys, date columns, LOINC codes
- **Transactions**: Wrap multi-statement operations (FHIR sync batches)

### Git Commits
- **Conventional Commits**: `feat:`, `fix:`, `docs:`, `test:`, `chore:`, `security:`
- **Scope**: Include feature or principle (e.g., `feat(fhir): add OAuth2 PKCE flow`)
- **Body**: Reference constitutional principle if applicable
- **Footer**: `Co-Authored-By: Claude <noreply@anthropic.com>` for AI-generated commits

## Medical Data Handling Rules

### FHIR Resources
- **Always preserve raw JSON**: Store in `raw_json TEXT` column alongside parsed fields
- **Validate before insert**: Use FHIR R4 spec validators (reject invalid resources)
- **LOINC codes**: Never strip or normalize (preserve exact code from source)
- **Reference ranges**: MANDATORY for all numeric Observation values
- **Timestamps**: ISO 8601 with timezone (use `date-fns` for parsing/formatting)

### DICOM Files
- **Read-only access**: NEVER modify DICOM files on disk
- **Metadata extraction**: Use pydicom, extract without loading pixel data for performance
- **Display transformations**: Window/level, zoom, pan - all in memory only
- **Annotations**: Store in SQLite `imaging_annotations` table (not embedded in DICOM)

### AI/ML Context
- **Medical data in prompts**: Always include source attribution (lab date, imaging study UID)
- **Citations required**: AI responses MUST reference specific data points
- **Graceful degradation**: If xAI API unavailable, show cached response or informative message
- **No PHI to API**: Sanitize data before sending (strip patient names, MRNs if present)

- **FHIR Token Refresh**: Race condition if multiple requests happen during refresh - use mutex lock
- **SQLite Performance**: VACUUM needed after bulk FHIR sync (>1000 observations) - run in background
